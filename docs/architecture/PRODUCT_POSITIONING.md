# InvoMatch — Product Positioning & Differentiation

## 1. Why This Product Exists

Most invoice reconciliation solutions already exist.
However, they are built for a different context:

- Enterprise-heavy environments
- Long onboarding cycles
- ERP-locked workflows
- Black-box or semi-explainable decisions

These solutions are not designed for:
- Fast-moving teams
- Mid-size companies
- Developer-led integrations
- Systems requiring full control and transparency

InvoMatch exists to solve this gap.

---

## 2. Target User (Strict Definition)

Primary segment:

- Companies processing 100–1000 invoices/month
- Using lightweight or mixed systems (not deep SAP/Oracle dependency)
- Need audit-ready reconciliation without enterprise overhead

Secondary segment:

- Fintech / accounting tools that need reconciliation as a module (API-first usage)

NOT targeting:

- Large enterprise AP automation
- Fully outsourced finance operations

---

## 3. Core Value Proposition

InvoMatch is:

A deterministic, explainable, run-based reconciliation system  
designed for control, speed, and integration

---

## 4. Differentiation (10x Angles)

### Explainability First
Every decision must include structured reasons.

### Run-Centered Architecture
Everything is a Reconciliation Run with full lifecycle.

### API-First
Designed for integration, not UI dependency.

### Lightweight Adoption
Fast setup, minimal configuration.

### Controlled Learning
User corrections → governed signals (not black-box ML)

---

## 5. First Real Use Case

User uploads:
- invoices
- payments

System:
- matches
- flags uncertain cases

User:
- reviews & corrects

System:
- learns & exports results

- End-to-end flow works
- Explainable decisions
